## backfill

The `temporal schedule backfill` command executes Actions ahead of their specified time range.
Backfilling can be used to fill in [Workflow Runs](/workflows#run-id) <span id="i-033ed803-ef56-4687-9ff5-1e04b6ade37a" class="clickable-i clickable-link-preview">i</span><div id="preview-modal-033ed803-ef56-4687-9ff5-1e04b6ade37a" class="preview-modal"><div class="modal-header"><div id="x-033ed803-ef56-4687-9ff5-1e04b6ade37a" class="clickable-x clickable-link-preview">x</div><b>Link preview:</b>&nbsp;&nbsp<a href="/workflows#run-id">/workflows#run-id</a></div><div class="preview-modal-title">What is a Run Id?</div><div class="preview-modal-description">A Run Id is a globally unique, platform-level identifier for a Workflow Execution.</div><div class="preview-modal-tags"><a class="preview-modal-tag" href="/tags/term">term</a> <a class="preview-modal-tag" href="/tags/explanation">explanation</a></div></div> from a time period when the Schedule was paused, or from before the Schedule was created.

Schedule backfills require a valid Schedule ID, along with the time in which to run the Schedule and a change to the overlap policy.

```
temporal schedule backfill --schedule-id 'your-schedule-id' \
--overlap-policy 'BufferAll' 				\
--start-time '2022-05-0101T00:00:00Z'		\
--end-time '2022-05-31T23:59:59Z'
```

## create

The `temporal schedule create` command creates a new [Schedule](/workflows#schedule) <span id="i-4f5624f8-9b67-4085-a467-e3b5703421a9" class="clickable-i clickable-link-preview">i</span><div id="preview-modal-4f5624f8-9b67-4085-a467-e3b5703421a9" class="preview-modal"><div class="modal-header"><div id="x-4f5624f8-9b67-4085-a467-e3b5703421a9" class="clickable-x clickable-link-preview">x</div><b>Link preview:</b>&nbsp;&nbsp<a href="/workflows#schedule">/workflows#schedule</a></div><div class="preview-modal-title">What is a Schedule?</div><div class="preview-modal-description">A Schedule enables the scheduling of Workflow Executions.</div><div class="preview-modal-tags"><a class="preview-modal-tag" href="/tags/term">term</a> <a class="preview-modal-tag" href="/tags/explanation">explanation</a></div></div>.
Newly created Schedules return a Schedule ID to be used in other Schedule commands.

Schedules need to follow a format like the example shown here:

```
temporal schedule create \
    --schedule-id 'your-schedule-id' \
    --workflow-id 'your-workflow-id' \
    --task-queue 'your-task-queue' \
    --workflow-type 'YourWorkflowType'
```

Any combination of `--calendar`, `--interval`, and `--cron` is supported.
Actions will be executed at any time specified in the Schedule.

The `temporal schedule toggle` command can pause and unpause a [Schedule](/workflows#schedule) <span id="i-31b286be-acef-4ac8-8205-34bbdbf337c8" class="clickable-i clickable-link-preview">i</span><div id="preview-modal-31b286be-acef-4ac8-8205-34bbdbf337c8" class="preview-modal"><div class="modal-header"><div id="x-31b286be-acef-4ac8-8205-34bbdbf337c8" class="clickable-x clickable-link-preview">x</div><b>Link preview:</b>&nbsp;&nbsp<a href="/workflows#schedule">/workflows#schedule</a></div><div class="preview-modal-title">What is a Schedule?</div><div class="preview-modal-description">A Schedule enables the scheduling of Workflow Executions.</div><div class="preview-modal-tags"><a class="preview-modal-tag" href="/tags/term">term</a> <a class="preview-modal-tag" href="/tags/explanation">explanation</a></div></div>.

Toggling a Schedule requires a reason to be entered on the command line.
Use `--reason` to note the issue leading to the pause or unpause.

Schedule toggles are passed in this format:
`temporal schedule toggle --schedule-id 'your-schedule-id' --pause --reason "paused because the database is down"`
`temporal schedule toggle --schedule-id 'your-schedule-id' --unpause --reason "the database is back up"`

## trigger

The `temporal schedule trigger` command triggers an immediate action with a given [Schedule](/workflows#schedule) <span id="i-1051e415-9472-446c-875b-c4cb063eea6c" class="clickable-i clickable-link-preview">i</span><div id="preview-modal-1051e415-9472-446c-875b-c4cb063eea6c" class="preview-modal"><div class="modal-header"><div id="x-1051e415-9472-446c-875b-c4cb063eea6c" class="clickable-x clickable-link-preview">x</div><b>Link preview:</b>&nbsp;&nbsp<a href="/workflows#schedule">/workflows#schedule</a></div><div class="preview-modal-title">What is a Schedule?</div><div class="preview-modal-description">A Schedule enables the scheduling of Workflow Executions.</div><div class="preview-modal-tags"><a class="preview-modal-tag" href="/tags/term">term</a> <a class="preview-modal-tag" href="/tags/explanation">explanation</a></div></div>.
By default, this action is subject to the Overlap Policy of the Schedule.

Schedule triggers are passed in this format:
`temporal schedule trigger` can be used to start a Workflow Run immediately.
`temporal schedule trigger --schedule-id 'your-schedule-id'`

The Overlap Policy of the Schedule can be overridden as well.
`temporal schedule trigger --schedule-id 'your-schedule-id' --overlap-policy 'AllowAll'`

## update

The `temporal schedule update` command updates an existing [Schedule](/workflows#schedule) <span id="i-d00619bc-fa4b-4e59-8a39-088d76f1fa4c" class="clickable-i clickable-link-preview">i</span><div id="preview-modal-d00619bc-fa4b-4e59-8a39-088d76f1fa4c" class="preview-modal"><div class="modal-header"><div id="x-d00619bc-fa4b-4e59-8a39-088d76f1fa4c" class="clickable-x clickable-link-preview">x</div><b>Link preview:</b>&nbsp;&nbsp<a href="/workflows#schedule">/workflows#schedule</a></div><div class="preview-modal-title">What is a Schedule?</div><div class="preview-modal-description">A Schedule enables the scheduling of Workflow Executions.</div><div class="preview-modal-tags"><a class="preview-modal-tag" href="/tags/term">term</a> <a class="preview-modal-tag" href="/tags/explanation">explanation</a></div></div>.

Like `temporal schedule create`, updated Schedules need to follow a certain format:

```
temporal schedule update 			    \
    --schedule-id 'your-schedule-id' 	\
    --workflow-id 'your-workflow-id' 	\
    --task-queue 'your-task-queue' 		\
    --workflow-type 'YourWorkflowType'
```

Updating a Schedule takes the given options and replaces the entire configuration of the Schedule with what's provided.
If you only change one value of the Schedule, be sure to provide the other unchanged fields to prevent them from being overwritten.
